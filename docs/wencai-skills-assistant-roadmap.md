# 问财 Skills 个人助手实施路线图

最后更新：2026-04-23

## 1. 文档定位

这份 roadmap 不再描述一套脱离代码的理想阶段，而是说明：

- 哪些能力已经完成
- 哪些能力正在收口
- 下一步应该先做什么
- 哪些技术升级是条件触发，而不是立即执行

优先级规则：

- 产品行为和契约细节以 `docs/wencai-skills-assistant-spec.md` 为准。
- 技术结构和演进边界以 `docs/wencai-skills-assistant-architecture.md` 为准。
- roadmap 负责排顺序，不重复维护另一套字段或实现定义。

## 3. 路线原则

后续迭代必须遵守：

1. 先收紧 contract，再扩功能。
2. 先修真实使用中的卡点，再做抽象升级。
3. 先保证浏览器里的链路顺，再考虑底层重构。
4. 不因为“未来可能需要”而提前引入数据库、Redis、队列或复杂分层。

## 4. 阶段划分

## 阶段 A：契约和工作台骨架

状态：已完成

目标：

- 固定产品边界
- 固定前后端 contract
- 固定工作台主布局

已交付：

- PRD
- Spec
- Architecture
- 工作台三栏布局
- 页面路由：`/`、`/templates`、`/watchlist`、`/settings`
- Pydantic / TypeScript 模型对齐

退出标准：

- 不再允许前后端各自维护一套字段定义
- 不再允许页面结构频繁改骨架

## 阶段 B：后端 MVP 主链路

状态：已完成

目标：

- 跑通自然语言提问到结构化结果

已交付：

- `GET /health`
- `GET /api/meta/status`
- `GET/POST/DELETE /api/sessions`
- `GET /api/sessions/{id}`
- `POST /api/chat`
- `POST /api/chat/follow-up`
- `POST /api/chat/compare`
- `GET/PUT /api/profile`
- 模板与候选池 CRUD
- 候选池股票解析：名称、6 位代码、带后缀代码均可识别并标准化
- 模式识别
- 路由策略
- 结果标准化
- JSON 持久化

关键结果：

- 能返回结构化 JSON
- 能流式返回 SSE
- 能保存消息快照
- 能基于快照继续追问

## 阶段 C：前端 MVP 可用性

状态：已完成

目标：

- 让工作台达到日常可用

已交付：

- 会话列表
- 消息流
- ChatComposer
- 结果侧栏
- 技能执行记录
- 追问建议
- 模板中心
- 候选池
- 候选池添加股票支持自动识别、预览、去重和错误提示
- 对话内加池：
  - 支持“把东阳光加入候选池”
  - 支持结果表 `☆` 一键加入候选池
- 设置页

关键结果：

- 用户可以在一个页面连续完成 提问 -> 看结果 -> 追问 -> 继续切换会话

## 阶段 D：真实使用体验收口

状态：进行中

目标：

- 从“功能齐全”收口到“个人使用顺手”

已完成项：

- 单股问法识别：`东阳光今天能买嘛`
- 单股买点问法识别：`东阳光建议多少价格买入`
- 操作建议卡：
  - 现在能不能追
  - 更好的买点
  - 失效条件
  - 止损/观察位
- 单股三周期分析：
  - 短线 / 中线 / 长线同时输出
  - 结合日 K、均线、MACD、RSI、布林带、量比和资金
- 会话关闭：
  - 左栏可直接关闭
  - 当前会话关闭后中栏清空
  - 后端归档隐藏而不是硬删除
- 开发态双地址可用：
  - `localhost:3000`
  - `127.0.0.1:3000`

仍需继续的事项：

1. 继续优化多周期文案的简洁度，避免 summary 过长。
2. 把浏览器回归脚本沉淀成可复用工具，而不是临时命令。
3. 完善异常态表现，让 API 失败时比现在更可解释。
4. 明确“关闭会话”和“恢复会话”是否要补二段式能力。
5. 继续校准 `auto -> medium/high/xhigh` 的分流命中率。

退出标准：

- 至少覆盖最常见的单股、筛选、追问、关闭会话链路
- 浏览器回归可以稳定复现
- 用户真实使用时不再频繁碰到“问得出来但答不顺”的问题

## 阶段 E：稳定性和可维护性增强

状态：下一阶段

目标：

- 让系统更稳、更容易维护，而不是先堆功能

优先事项：

1. 增加最小自动化测试集
2. 增加核心浏览器回归脚本
3. 梳理 `chat_engine.py` 的模块边界
4. 补充失败重试和降级策略
5. 让来源说明、技能原因、结果快照更容易排查

推荐任务包：

### 包 E1：自动化回归

- API smoke tests
- 会话关闭回归
- 单股买点回归
- `localhost` / `127.0.0.1` 双地址回归

### 包 E2：错误处理

- 空结果文案统一
- API 异常提示统一
- skill 失败时的降级路径

### 包 E3：代码结构收口

- 将过重的单文件逻辑拆出清晰职责边界
- 保持拆分后 contract 不变

退出标准：

- 主要回归不再依赖手工点一遍
- 出问题时能更快定位是 contract、数据源还是渲染问题

## 阶段 F：数据层升级

状态：条件触发

这一阶段不是默认下一步，只有当 JSON 存储开始成为问题时才启动。

触发条件：

- 会话和消息量增长明显拖慢读取
- 候选池需要复杂查询
- 需要更强的数据可靠性和迁移能力
- 需要多端同步

优先顺序：

1. SQLite
2. PostgreSQL

约束：

- 升级数据层不能改变现有 API contract
- 必须先保留 `repository` 作为稳定边界
- 迁移时先做等价替换，不顺手改业务语义

## 阶段 G：缓存和性能层升级

状态：条件触发

只有在以下问题真实出现后再做：

- 外部查询明显慢
- 同类查询重复率高
- 需要跨进程共享缓存

候选内容：

- 查询结果缓存
- 结构化结果缓存
- profile / 模板缓存

当前不作为默认排期。

## 5. 未来功能池

这些是可做项，但不进入最近一轮主优先级：

- 会话恢复 UI
- 候选池打分
- 个股比较评分卡
- 自定义编排规则
- 模拟炒股联动
- 每日盘前清单
- 盘后复盘草稿

进入排期前提：

- 当前工作台主链路已经稳定
- contract 不再频繁变化
- 真实使用里确实反复碰到该需求

## 6. 不做清单

最近阶段不做：

- 多人协作
- 权限系统
- 实盘交易执行
- 大而全的投研数据库
- 为了“工程完整度”强行上微服务
- 没有触发条件时提前接 Redis / PostgreSQL / SQLAlchemy

## 7. 近期待办优先级

如果按实际收益排序，下一步建议是：

1. 自动化浏览器回归脚本
2. 更自然的单股投资建议文案
3. 异常态和降级文案收口
4. `chat_engine.py` 内部职责拆分
5. 会话恢复能力评估

## 8. 完成定义

某个阶段只有同时满足以下条件，才算完成：

- spec 已同步
- architecture / roadmap 已同步
- 前后端 contract 已同步
- 前端构建通过
- 至少完成一轮关键回归

如果只是代码改了但文档没跟上，不算完成。
